    Authentication Architecture

    FlowPilot uses Firebase Authentication as the identity provider for production deployments. For local development, Keycloak can optionally be used. All services validate bearer tokens using Firebase Admin SDK (production) or JWKS (local dev).

    Authentication Flows

    Desktop Client Authentication

    Flow: Firebase Authentication SDK

    The desktop client (macOS app) uses the Firebase Authentication SDK with email/password sign-in.

    Steps:

    1. User initiates login - User enters credentials in app
    2. App calls Firebase Auth - Using Firebase iOS SDK
    3. Firebase validates credentials - Server-side validation
    4. Firebase returns ID token - JWT with uid, email, and custom claims
    5. App uses token for API calls - Bearer token in Authorization header

    Benefits:

    • No browser redirect required
    • Built-in email verification
    • Password reset functionality
    • Multi-factor authentication (MFA) support
    • Automatic token refresh

    Service-to-Service Authentication

    Flow: Google Cloud Identity Tokens

    Services authenticate to each other using Google Cloud identity tokens (Cloud Run).

    Service Account:

    • Automatic on Cloud Run
    • Uses Application Default Credentials (ADC)
    • Subject: Service account email
    • Persona: service or ai-agent

    Implementation Pattern (Production/GCP):

    import google.auth
import google.auth.transport.requests
from google.auth import compute_engine

def get_service_token():
    """Get service-to-service access token."""
    # On Cloud Run, this automatically uses the service account
    credentials, project = google.auth.default()

    # Get an ID token for service-to-service auth
    auth_req = google.auth.transport.requests.Request()
    credentials.refresh(auth_req)

    return credentials.token

# Use token in requests
headers = {"Authorization": f"Bearer {get_service_token()}"}
response = requests.post(url, json=data, headers=headers)

    Local Development:

    For local testing, services can use Firebase Admin SDK to create custom tokens or use test tokens from Firebase console.

    Token Structure

    FlowPilot uses minimal bearer tokens that carry only essential information.

    Token Claims

    Standard Claims:

    • uid - Firebase user ID - Pseudonymous identifier (mapped to sub in services)
    • iss - Issuer (Firebase project URL)
    • aud - Audience (Firebase project ID)
    • exp - Expiry timestamp
    • iat - Issued-at timestamp
    • auth_time - Authentication timestamp
    • email - User email (if available)
    • email_verified - Email verification status

    Custom Claims:

    • persona - Business role (e.g., traveler, travel-agent, ai-agent)

    What Tokens Do NOT Contain

    No PII:

    • ❌ No names
    • ❌ No email addresses
    • ❌ No phone numbers
    • ❌ No personal preferences
    • ❌ No consent details
    • ❌ No profile attributes

    Benefits:

    • Tokens remain small and stable
    • Privacy preserved by design
    • Lower breach impact
    • GDPR-friendly

    Example Token Payload

    {
  "uid": "89eb5366-bab3-46e4-b8e1-abc5f2ea4631",
  "email": "user@example.com",
  "email_verified": true,
  "persona": "traveler",
  "iss": "https://securetoken.google.com/vision-course-476214",
  "aud": "vision-course-476214",
  "exp": 1704901200,
  "iat": 1704897600,
  "auth_time": 1704897600
}

    Token Validation

    All services validate tokens using Firebase Admin SDK.

    Firebase Admin SDK Validation

    Benefits:

    • Built-in verification - Handles signature validation automatically
    • High performance - Public keys cached by SDK
    • Resilient - No dependency on Firebase servers for validation
    • Scalable - Token verification is local
    • Automatic key rotation - SDK handles key updates

    Validation Steps

    1. Extract token from Authorization header
    2. Initialize Firebase Admin SDK (if not already initialized)
    3. Call verify_id_token() - SDK handles all validation:
    4. Verifies signature using Firebase public keys
    5. Validates issuer (must be Firebase)
    6. Validates audience (must match project ID)
    7. Checks token expiry (exp > now)
    8. Validates issued-at time (iat)
    9. Extract claims from decoded token:
    10. Map uid to sub for consistency
    11. Extract custom claims (including persona)
    12. Return validated claims

    Implementation Pattern

    All services use the shared security library:

    from fastapi import Depends, HTTPException
import security

def get_token_claims(
    token_claims: dict = Depends(security.verify_token)
) -> dict:
    """Dependency that validates JWT and returns claims."""
    return token_claims

@app.post("/protected-endpoint")
def protected_handler(
    token_claims: dict = Depends(get_token_claims)
):
    user_sub = token_claims["sub"]
    user_persona = token_claims.get("persona")

    # Use validated claims
    return {"message": f"Hello {user_sub}"}

    Validation Configuration

    Environment Variables:

    Firebase Admin SDK uses Application Default Credentials (ADC) on Cloud Run - no explicit configuration needed for production.

    For local development:

    # Optional: Path to service account key
GOOGLE_APPLICATION_CREDENTIALS=/path/to/serviceAccountKey.json

    Firebase Configuration

    Firebase Project

    FlowPilot uses Firebase Authentication in the GCP project vision-course-476214.

    Setup Steps:

    1. Enable Firebase in Firebase Console
    2. Enable Authentication → Sign-in methods → Email/Password
    3. Enable Firestore Database (Native mode)

    User Data Model

    Firebase Authentication:

    Stores core user identity:

    • uid - Firebase user ID (pseudonymous)
    • email - User email address
    • displayName - Optional display name
    • emailVerified - Verification status

    Firestore (users collection):

    Stores user profile and preferences:

    {
  "persona": ["traveler"],
  "autobook_consent": "Yes",
  "autobook_price": "1500",
  "autobook_leadtime": "7",
  "autobook_risklevel": "2"
}

    Custom Claims (Optional):

    Can be set on Firebase Auth users for inclusion in ID tokens:

    {
  "persona": ["traveler", "travel-agent"]
}

    Personas

    Users can have multiple personas:

    Supported Personas:

    • traveler - End users booking travel
    • travel-agent - Human travel agents
    • office-manager - Office administrators
    • booking-assistant - Booking support staff
    • ai-agent - AI agents (service accounts)

    Persona Storage:

    1. Primary storage: Firestore users collection
    2. Optional: Firebase custom claims (for token inclusion)
    3. Services read persona from token or Firestore

    User Provisioning

    Users can be created via Firebase Admin SDK:

    from firebase_admin import auth, firestore

# Create user in Firebase Auth
user = auth.create_user(
    email='user@example.com',
    password='secure-password',
    display_name='Test User'
)

# Set custom claims
auth.set_custom_user_claims(user.uid, {
    'persona': ['traveler']
})

# Create Firestore profile
db = firestore.client()
db.collection('users').document(user.uid).set({
    'persona': ['traveler'],
    'autobook_consent': 'Yes',
    'autobook_price': '1500',
    'autobook_leadtime': '7',
    'autobook_risklevel': '2'
})

    Setup Scripts:

    • scripts/setup_firebase_user.py - Create test users
    • flowpilot-provisioning/seed_firebase_users.py - Bulk user provisioning

    TLS and Certificates

    Production (Firebase/GCP)

    Firebase and Cloud Run handle TLS automatically:

    • Firebase uses Google's managed certificates
    • Cloud Run provides automatic HTTPS
    • No certificate management required

    Local Development (Optional Keycloak)

    If using Keycloak for local development, use mkcert:

    Setup:

    # Install mkcert
brew install mkcert

# Install local CA
mkcert -install

# Copy root CA to project
cp "$(mkcert -CAROOT)/rootCA.pem" infra/certs/mkcert-rootCA.pem

    Token Lifecycle

    Token Issuance

    1. User authenticates via Firebase Authentication SDK
    2. Firebase validates credentials server-side
    3. Firebase generates JWT with claims and custom claims
    4. Firebase signs JWT with Firebase private keys
    5. Client receives ID token (and refresh token)

    Token Usage

    1. Client includes token in Authorization header
    2. Service extracts token from header
    3. Service validates token using JWKS
    4. Service extracts claims for authorization
    5. Service processes request based on claims

    Token Expiry

    Access Token:

    • Short-lived (configurable, default: 5 minutes)
    • Must be refreshed or re-authenticated

    Refresh Token:

    • Longer-lived (configurable, default: 30 minutes)
    • Used to obtain new access tokens
    • Not used in service-to-service flow

    Token Refresh

    Desktop Client:

    Clients can refresh tokens using the refresh token:

    response = requests.post(
    keycloak_token_url,
    data={
        "grant_type": "refresh_token",
        "refresh_token": refresh_token,
        "client_id": client_id,
    }
)
new_access_token = response.json()["access_token"]

    Service Accounts:

    Service accounts request new tokens as needed (no refresh token).

    Security Considerations

    Token Storage

    Desktop Client:

    • Store tokens securely (macOS Keychain)
    • Never log tokens
    • Clear tokens on logout

    Services:

    • Tokens are transient (not stored)
    • Service accounts fetch fresh tokens as needed
    • Never log tokens

    Token Transmission

    • Always use HTTPS in production
    • Use Authorization header (not query parameters)
    • Never expose tokens in logs or error messages

    Token Validation

    • Always validate signature using JWKS
    • Always check expiry (exp claim)
    • Always verify issuer (prevent token substitution)
    • Consider audience validation for multi-tenant scenarios

    JWKS Caching

    The shared security library caches JWKS keys:

    • Cache TTL: 1 hour (configurable)
    • Refresh on key miss - If kid not in cache
    • Automatic rotation - Supports key rotation

    Troubleshooting

    Invalid Token Signature

    Symptom: 401 Unauthorized with "Invalid token signature"

    Causes:

    • JWKS endpoint unreachable
    • Token signed with wrong key
    • Clock skew between services

    Solutions:

    # Check JWKS endpoint
curl https://localhost:8443/realms/flowpilot/protocol/openid-connect/certs --insecure

# Check service logs
docker compose logs flowpilot-authz-api | grep -i jwks

# Restart service to refresh JWKS cache
docker compose restart flowpilot-authz-api

    Token Expired

    Symptom: 401 Unauthorized with "Token expired"

    Solution: Client must refresh token or re-authenticate

    Keycloak Not Ready

    Symptom: Services fail to start with JWKS errors

    Solution:

    # Wait for Keycloak to fully start (60+ seconds)
docker compose logs -f keycloak
# Look for: "Listening on: https://0.0.0.0:8443"

    Wrong Issuer

    Symptom: 401 Unauthorized with "Invalid issuer"

    Cause: KEYCLOAK_ISSUER environment variable doesn't match token iss claim

    Solution: Verify issuer configuration matches Keycloak realm URL

    Best Practices

    Development

    • ✅ Use mkcert for local TLS
    • ✅ Use short-lived tokens for testing
    • ✅ Test token expiry scenarios
    • ✅ Never commit tokens or secrets

    Production

    • ✅ Use proper TLS certificates
    • ✅ Use managed OIDC provider or HA Keycloak
    • ✅ Rotate client secrets regularly
    • ✅ Monitor authentication failures
    • ✅ Implement token refresh logic
    • ✅ Set appropriate token lifespans
    • ❌ Never use verify=False for TLS
    • ❌ Never log tokens or secrets
    • ❌ Never store tokens in plaintext
