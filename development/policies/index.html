    Policy Development Guide

    This guide explains how FlowPilot's authorization policies work, how they're constructed, and the role of the authz-api in policy evaluation.

    Overview

    FlowPilot uses Open Policy Agent (OPA) with Rego policies for declarative access control. The policy engine answers the question:

    "Is this subject allowed to perform this action on this resource?"

    Architecture

    Policy Decision Flow

    1. PEP (Policy Enforcement Point) - Application service (e.g., ai-agent-api)
    2. Authz-API - Authorization façade (enriches requests, calls PIPs)
    3. Delegation-API - Policy Information Point for ReBAC
    4. OPA - Policy Decision Point (evaluates Rego policies)
    PEP → Authz-API → Delegation-API (ReBAC)
                 → OPA (ABAC)

    The Role of Authz-API

    The authz-api acts as the orchestration layer between PEPs and the policy engine. It:

    1. Validates JWTs - Ensures bearer tokens are valid
    2. Resolves Delegation - Queries delegation-api for ReBAC data
    3. Enriches Context - Adds policy-relevant attributes (consent, thresholds)
    4. Normalizes Data - Converts types (dates to RFC3339, strings to numbers)
    5. Calls OPA - Sends enriched input to policy engine
    6. Returns Decision - Structured allow/deny with reason codes

    Key Insight: The authz-api ensures OPA receives decision-ready facts, not raw requests.

    The Auto-Book Policy

    FlowPilot's primary policy is auto_book.rego, which controls autonomous AI booking and delegated access.

    Policy File Location

    infra/opa/policies/auto_book.rego

    Supported Actions

    • execute - Execute workflow items (book travel)
    • read - View workflows (read-only access)

    Policy Construction: Step by Step

    Step 1: Package and Default

    package auto_book

default allow := false

    What this does:

    • Declares the policy package name
    • Fail-closed by default - Deny unless explicitly allowed

    Authz-API role: Interprets allow = false as a deny decision

    Step 2: Action Routing

    allow if {
  input.action.name == "execute"
  allow_execute
}

allow if {
  input.action.name == "read"
  allow_read
}

    What this does:

    • Routes to different rule sets based on the requested action
    • execute requires full authorization gates
    • read has simpler requirements

    Authz-API role: Includes action.name in the OPA input based on the PEP's request

    Step 3: Execute Gates

    allow_execute if {
  authorized_principal  # Anti-spoofing and delegation check
  persona_valid         # Persona check
  has_consent          # User consent
  acceptable_risk      # Risk threshold
  within_cost_limit    # Cost ceiling
  sufficient_advance   # Lead time requirement
}

    What this does:

    • Defines 6 gates that all must pass for execution
    • Each gate is a separate rule evaluated independently
    • If any gate fails, allow_execute is false

    Authz-API role: Provides all data needed for gate evaluation (more below)

    Step 4: Anti-Spoofing Check

    # Owner directly accessing their own resource
authorized_principal if {
  input.subject.persona != "ai-agent"
  principal_id := input.subject.id
  owner_id := input.resource.properties.owner.id
  principal_id == owner_id
}

# AI-agent activated by the owner
authorized_principal if {
  input.subject.persona == "ai-agent"
  principal_id := input.context.principal.id
  owner_id := input.resource.properties.owner.id
  principal_id == owner_id
}

# AI-agent in autonomous mode
authorized_principal if {
  input.subject.persona == "ai-agent"
  input.resource.properties.owner.autobook_consent == true
  not input.context.delegation.valid
}

# Valid delegation
authorized_principal if {
  has_valid_delegation_for_action
}

    What this does:

    • Prevents principal spoofing (acting as someone else without permission)
    • Allows 4 scenarios:
    • Owner acting directly - User executes their own workflow
    • Owner via agent - User activates AI agent to execute
    • Autonomous AI - AI acts independently with consent
    • Delegated user - Another user with explicit delegation

    Authz-API role:

    • Queries delegation-api to get context.delegation.valid
    • Includes context.delegation.delegated_actions array
    • Ensures context.principal is set when subject ≠ owner

    Delegation Check Helper:

    has_valid_delegation_for_action if {
  input.context.delegation.valid == true
  input.action.name in input.context.delegation.delegated_actions
}

    This checks if: - Delegation exists (valid == true) - Delegation grants the requested action (e.g., execute)

    Step 5: Persona Validation

    valid_agent_personas := {"travel-agent", "ai-agent", "office-manager", "booking-assistant"}

# AI-agent activated by delegated user
persona_valid if {
  input.subject.persona == "ai-agent"
  principal_id := input.context.principal.id
  owner_id := input.resource.properties.owner.id
  principal_id != owner_id
  principal_persona := input.context.principal.persona
  principal_persona != ""
  valid_agent_personas[principal_persona]
}

# AI-agent activated by owner
persona_valid if {
  input.subject.persona == "ai-agent"
  principal_id := input.context.principal.id
  owner_id := input.resource.properties.owner.id
  principal_id == owner_id
  owner_persona := input.resource.properties.owner.persona
  principal_persona := input.context.principal.persona
  principal_persona == owner_persona
}

# Human user with delegation
persona_valid if {
  input.subject.persona != "ai-agent"
  principal_id := input.subject.id
  owner_id := input.resource.properties.owner.id
  principal_id != owner_id
  input.subject.persona != ""
  valid_agent_personas[input.subject.persona]
}

# Owner executing directly
persona_valid if {
  input.subject.persona != "ai-agent"
  principal_id := input.subject.id
  owner_id := input.resource.properties.owner.id
  principal_id == owner_id
  owner_persona := input.resource.properties.owner.persona
  input.subject.persona == owner_persona
}

    What this does:

    • Validates that the user's business role (persona) is appropriate
    • Delegated users must have agent personas (travel-agent, etc.)
    • Owners must match their own persona

    Authz-API role:

    • Extracts persona from JWT custom claims
    • Fetches owner.persona from user profile (Firestore)
    • Includes both in OPA input for comparison

    Step 6: Consent Check

    has_consent if {
  input.resource.properties.owner.autobook_consent == true
}

    What this does:

    • Requires explicit user opt-in for autonomous booking
    • Consent is per-user, not per-workflow

    Authz-API role:

    • Fetches autobook_consent from user profile (Firestore)
    • Converts string ("Yes"/"No") to boolean
    • Includes in resource.properties.owner

    Step 7: Cost Limit Check

    within_cost_limit if {
  planned_price := input.resource.properties.planned_price
  max_cost := input.resource.properties.owner.autobook_price
  planned_price <= max_cost
}

    What this does:

    • Ensures booking cost doesn't exceed user's configured limit
    • Cost limit is per-user preference

    Authz-API role:

    • Extracts planned_price from workflow item resource
    • Fetches autobook_price from user profile
    • Converts both to numbers for comparison

    Step 8: Lead Time Check

    sufficient_advance if {
  departure_date_str := input.resource.properties.departure_date
  departure := time.parse_rfc3339_ns(departure_date_str)
  now := time.now_ns()
  min_days := input.resource.properties.owner.autobook_leadtime
  delta_days := (departure - now) / 1000000000 / 60 / 60 / 24
  delta_days >= min_days
}

    What this does:

    • Requires minimum advance notice before departure
    • Prevents last-minute autonomous bookings

    Authz-API role:

    • Converts departure_date to RFC3339 format (OPA requirement)
    • Fetches autobook_leadtime from user profile
    • Ensures departure_date is a valid timestamp

    Step 9: Risk Check

    acceptable_risk if {
  not input.resource.properties.airline_risk_score
}

acceptable_risk if {
  risk := input.resource.properties.airline_risk_score
  max_risk := input.resource.properties.owner.autobook_risklevel
  risk <= max_risk
}

    What this does:

    • Checks airline risk score against user's tolerance
    • Allows execution if risk score is missing (optional field)
    • Denies if risk exceeds configured threshold

    Authz-API role:

    • Includes airline_risk_score if present in workflow item
    • Fetches autobook_risklevel from user profile
    • Converts both to numbers

    Step 10: Reason Codes

    reasons[code] if {
  input.action.name == "execute"
  principal_id := input.subject.id
  owner_id := input.resource.properties.owner.id
  principal_id != owner_id
  not input.context.delegation.valid
  code := "auto_book.principal_spoofing"
}

reasons[code] if {
  input.action.name == "execute"
  code := "auto_book.no_consent"
  authorized_principal
  not has_consent
}

# ... more reason rules ...

    What this does:

    • Provides structured reason codes for denials
    • Each failed gate has a corresponding reason
    • Supports debugging and audit trails

    Authz-API role:

    • Collects reasons array from OPA response
    • Returns reason codes to PEP for logging/debugging
    • Maps codes to human-readable messages

    OPA Input Structure

    Here's what the authz-api sends to OPA (enriched from PEP's AuthZEN request):

    {
  "subject": {
    "id": "ai-agent",
    "persona": "ai-agent"
  },
  "action": {
    "name": "execute"
  },
  "resource": {
    "type": "workflow_item",
    "id": "i_abc123",
    "properties": {
      "workflow_id": "w_xyz789",
      "planned_price": 500.0,
      "departure_date": "2026-01-30T00:00:00Z",
      "airline_risk_score": 7.0,
      "owner": {
        "id": "user-uuid",
        "persona": "traveler",
        "autobook_consent": true,
        "autobook_price": 1500,
        "autobook_leadtime": 7,
        "autobook_risklevel": 10
      }
    }
  },
  "context": {
    "delegation": {
      "valid": true,
      "delegation_chain": ["user1", "user2"],
      "delegated_actions": ["read", "execute"]
    },
    "principal": {
      "id": "user-uuid",
      "persona": "traveler"
    }
  }
}

    Data Sources for OPA Input

    Field Source Provider
    subject.id AuthZEN request PEP
    subject.persona JWT custom claims Authz-API
    action.name AuthZEN request PEP
    resource.id AuthZEN request PEP
    resource.properties.planned_price Workflow item data PEP
    resource.properties.departure_date Workflow item data PEP (normalized by authz-api)
    resource.properties.owner.autobook_* User profile (Firestore) Authz-API
    context.delegation.* Delegation graph Delegation-API (via authz-api)
    context.principal AuthZEN request or JWT PEP / Authz-API

    Authz-API Enrichment Process

    1. Validate JWT

    # Validate bearer token using Firebase Admin SDK
token_claims = security.verify_token(credentials)
user_sub = token_claims["sub"]

    2. Query Delegation-API

    # Check if delegation exists
delegation_response = requests.post(
    f"{DELEGATION_API_BASE_URL}/v1/delegations/validate",
    json={
        "principal_id": principal_id,
        "delegate_id": subject_id,
        "workflow_id": workflow_id,
        "action": action_name,
    },
    headers=headers
)

# Add to context
input_data["context"]["delegation"] = {
    "valid": delegation_response.json()["valid"],
    "delegation_chain": delegation_response.json().get("delegation_chain", []),
    "delegated_actions": delegation_response.json().get("delegated_actions", []),
}

    3. Fetch User Profile

    # Fetch owner's profile attributes from Firestore
owner_attributes = profile.fetch_attributes(owner_id)

# Add to resource
input_data["resource"]["properties"]["owner"] = {
    "id": owner_id,
    "persona": owner_attributes.get("persona", ["traveler"])[0],
    "autobook_consent": owner_attributes.get("autobook_consent") == "Yes",
    "autobook_price": int(owner_attributes.get("autobook_price", "0")),
    "autobook_leadtime": int(owner_attributes.get("autobook_leadtime", "0")),
    "autobook_risklevel": int(owner_attributes.get("autobook_risklevel", "0")),
}

    4. Normalize Data

    # Convert departure date to RFC3339
if "departure_date" in properties:
    # Convert YYYY-MM-DD to YYYY-MM-DDTHH:MM:SSZ
    departure_date = properties["departure_date"]
    properties["departure_date"] = f"{departure_date}T00:00:00Z"

# Ensure numeric types
properties["planned_price"] = float(properties.get("planned_price", 0))
if "airline_risk_score" in properties:
    properties["airline_risk_score"] = float(properties["airline_risk_score"])

    5. Call OPA

    opa_response = requests.post(
    f"{OPA_URL}/v1/data/auto_book/allow",
    json={"input": input_data},
    headers={"Content-Type": "application/json"},
)

allow = opa_response.json().get("result", False)

    6. Return Decision

    return {
    "decision": allow,
    "reasons": reasons,  # From OPA
    "context": {
        "delegation_valid": delegation_valid,
        "policy_evaluated": True,
    }
}

    Common Policy Patterns

    Pattern 1: Fail-Closed

    default allow := false

allow if {
  # Explicit conditions
}

    Always deny unless explicitly allowed.

    Pattern 2: Multiple Allow Rules

    allow if { condition_a }
allow if { condition_b }

    Any rule can grant access (OR logic).

    Pattern 3: Gated Allow

    allow if {
  gate_1
  gate_2
  gate_3
}

    All gates must pass (AND logic).

    Pattern 4: Reason Tracking

    reasons[code] if {
  not some_condition
  code := "policy.failure_reason"
}

    Provide structured denial reasons.

    Best Practices

    For Policy Authors

    1. Fail-closed - Default to deny
    2. Explicit rules - Make allow conditions clear
    3. Reason codes - Provide debugging information
    4. Type safety - Validate data types in Rego
    5. Comments - Document complex logic

    For Authz-API Developers

    1. Normalize data - Convert types before OPA
    2. Validate inputs - Ensure required fields exist
    3. Enrich completely - Provide all data OPA needs
    4. Handle errors - Graceful degradation on PIP failures
    5. Log decisions - Audit all authorization outcomes

    For PEP Developers

    1. Minimal requests - Send only intent, not policy data
    2. Consistent format - Use AuthZEN structure
    3. Include context - Provide principal when subject ≠ owner
    4. Enforce decisions - Always respect authz-api response

    Deployment

    Local Development

    Policies are mounted from filesystem with hot-reload:

    # docker-compose.yml
opa:
  volumes:
    - ./infra/opa/policies:/policies

    Production (GCP Cloud Run)

    Policies are baked into the OPA container image:

    # infra/opa/Dockerfile
COPY infra/opa/policies /policies

    To update policies in production:

    1. Edit infra/opa/policies/auto_book.rego
    2. Rebuild OPA container
    3. Push to Artifact Registry
    4. Deploy to Cloud Run
    docker build -f infra/opa/Dockerfile -t us-central1-docker.pkg.dev/PROJECT/flowpilot/opa:latest .
docker push us-central1-docker.pkg.dev/PROJECT/flowpilot/opa:latest
gcloud run deploy flowpilot-opa --image us-central1-docker.pkg.dev/PROJECT/flowpilot/opa:latest

    Summary

    Policy Layer (OPA): - Declarative rules in Rego - Pure policy logic, no data fetching - Receives decision-ready input

    Orchestration Layer (Authz-API): - Validates JWTs - Queries delegation-api (ReBAC) - Fetches user profiles (ABAC attributes) - Normalizes and enriches data - Calls OPA with complete context

    Enforcement Layer (PEP): - Submits AuthZEN requests - Enforces authz-api decisions - Never evaluates policy directly

    This separation keeps policies clean, authorization centralized, and applications simple.
