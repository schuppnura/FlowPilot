    Persona Guide

    This guide explains FlowPilot's persona system, including the data model, API operations, authorization rules, and technical implementation details.

    Overview

    A persona represents a business role that a user assumes in a specific context. Personas are the foundation of FlowPilot's authorization model and drive policy decisions for both autonomous AI booking and delegated access.

    Key Concepts:

    • A single user may have multiple personas (e.g., both traveler and travel-agent)
    • Each authorization request evaluates against exactly one active persona
    • Personas include autobook preferences (consent, cost limits, lead time, risk tolerance)
    • Personas have temporal validity (valid_from, valid_till) and status (active, inactive, suspended)
    • Personas are user-owned and independently configurable

    By introducing the concept persona, authorisation policies can rely on attributes that reflect the business role or mandate that a person plays at that time. A FlowPilot persona embodies attributes that are specific for that business role or mandate and can be self-declared. With its persona-driven authorisation policies, FlowPilot: 1. Enables User Lifecycle Management to be done at the persona level rather than (traditionally) at the user account level. This means that when a person leaves a function, their account(s) do not need to be deactivated: only the associated persona needs to be deactivated. It also allows people to already start using their account, even before one of their personas has been activated. 2. Enables Delegated Security Administration whereby security admin (indeed!) is moved to people close to the end user. This is in contrast to having a sysadmin assign permissions, roles and groups. Delegating security administration to the people themselves, fosters timeliness, accuracy, appropriateness, and freshness. 3. Enables effortless implementation of Segregation of Duties, by having different personas corresponding to different duties. 4. Enables a policy-driven implementation of User Managed Access. FlowPilot personas bring the management of access control to a new level: it easily combines the benefits of RBAC and ABAC while simplifying its administration. In fact, delegating persona administration using FlowPilot allows access to be controlled at massive scales: no longer is a central sysadmin group needed to assign permissions, access rights, roles, and groups to end users. Instead, delegated persona administration puts people in the centre of their own administration: they manage their own attributes and personas and therefor their own access.

    Persona Data Model

    Core Attributes

    Field Type Description
    persona_id UUID Unique identifier for the persona
    user_sub String Owner's user ID (Firebase UID or Keycloak sub)
    title String Persona title (e.g., "traveler", "travel-agent")
    status String Current status: "active", "inactive", "suspended", "expired"
    scope Array[String] Allowed actions (e.g., ["read", "execute"])
    valid_from ISO 8601 Timestamp when persona becomes active
    valid_till ISO 8601 Timestamp when persona expires
    created_at ISO 8601 Creation timestamp
    updated_at ISO 8601 Last modification timestamp

    Autobook Attributes

    Autobook attributes define autonomous AI booking preferences:

    Field Type Description
    autobook_consent Boolean Whether user allows autonomous AI booking
    autobook_price Integer Maximum booking price (EUR)
    autobook_leadtime Integer Minimum advance booking time (days)
    autobook_risklevel Integer Maximum airline risk score (0-100)

    Example Persona Document

    {
  "persona_id": "b9678f30-f4b0-4033-82db-846357311165",
  "user_sub": "PKbHpCqDnLcNywEo8pev8yQmoU43",
  "title": "traveler",
  "status": "active",
  "scope": ["read", "execute"],
  "valid_from": "2024-01-01T00:00:00Z",
  "valid_till": "2026-12-31T23:59:59Z",
  "created_at": "2026-01-13T17:00:00Z",
  "updated_at": "2026-01-13T17:00:00Z",
  "autobook_consent": true,
  "autobook_price": 10000,
  "autobook_leadtime": 7,
  "autobook_risklevel": 5
}

    Supported Persona Titles

    FlowPilot defines the following persona titles for the "Travel" use case:

    End-User Personas

    • traveler - End users who book travel for themselves

      • Can own workflows
      • Can delegate to agents
      • Autobook preferences apply to their workflows

    • business-traveler - End users who book travel for business

      • Can own workflows
      • Can delegate to agents
      • Autobook preferences apply to their workflows

    Delegation Personas

    • travel-agent - Professional travel agents

      • Can execute workflows on behalf of travelers (with delegation)
      • Trusted role for booking operations

    • office-manager - Office administrators

      • Can manage travel bookings for employees
      • Can execute workflows with delegation

    • booking-assistant - Booking support staff

      • Can assist with travel bookings
      • Can execute workflows with delegation

    System Personas

    • ai-agent - Autonomous AI agents
      • System-only persona (not stored in user-profile-api)
      • Used by ai-agent-api and domain-services-api
      • Subject to autobook policy gates

    Persona Lifecycle

    1. Creation

    Personas are created via the user-profile-api POST /v1/personas endpoint.

    Request: 

    POST /v1/personas
Authorization: Bearer <user-token>
Content-Type: application/json

{
  "title": "traveler",
  "scope": ["read", "execute"],
  "valid_from": "2024-01-01T00:00:00Z",
  "valid_till": "2026-12-31T23:59:59Z",
  "status": "active",
  "autobook_consent": true,
  "autobook_price": 5000,
  "autobook_leadtime": 7,
  "autobook_risklevel": 3
}

    Response: 

    {
  "persona_id": "b9678f30-f4b0-4033-82db-846357311165",
  "user_sub": "PKbHpCqDnLcNywEo8pev8yQmoU43",
  "title": "traveler",
  "status": "active",
  "scope": ["read", "execute"],
  "valid_from": "2024-01-01T00:00:00Z",
  "valid_till": "2026-12-31T23:59:59Z",
  "created_at": "2026-01-13T17:00:00Z",
  "updated_at": "2026-01-13T17:00:00Z",
  "autobook_consent": true,
  "autobook_price": 5000,
  "autobook_leadtime": 7,
  "autobook_risklevel": 3
}

    Authorization: The user must be authenticated. The persona is created for the authenticated user (extracted from JWT sub claim).

    2. Listing

    Users can list their own personas:

    GET /v1/personas?status=active
Authorization: Bearer <user-token>

    Service accounts can list personas for any user:

    GET /v1/users/{user_sub}/personas?status=active
Authorization: Bearer <service-token>

    This endpoint is used by authz-api to fetch persona data for authorization decisions.

    3. Retrieval

    Fetch a specific persona by ID:

    GET /v1/personas/{persona_id}
Authorization: Bearer <user-token>

    Authorization: The persona must belong to the authenticated user.

    4. Update

    Modify persona attributes:

    PUT /v1/personas/{persona_id}
Authorization: Bearer <user-token>
Content-Type: application/json

{
  "autobook_price": 8000,
  "autobook_leadtime": 14,
  "status": "active"
}

    All fields are optional (partial update). Only provided fields are updated.

    Authorization: The persona must belong to the authenticated user.

    5. Deletion

    Delete a persona:

    DELETE /v1/personas/{persona_id}
Authorization: Bearer <user-token>

    Authorization: The persona must belong to the authenticated user.

    Technical Implementation

    Storage Backends

    FlowPilot supports two storage backends for personas:

    Firestore (GCP Production)

    Used by Cloud Run deployment with Firebase Authentication: - Serverless, scalable NoSQL database - Collection: personas - Document ID = persona_id - Requires composite indexes for efficient queries

    Required Firestore Indexes:

    # Index 1: user_sub + status + created_at (descending)
gcloud firestore indexes composite create \
  --collection-group=personas \
  --query-scope=COLLECTION \
  --field-config field-path=user_sub,order=ascending \
  --field-config field-path=status,order=ascending \
  --field-config field-path=created_at,order=descending

# Index 2: user_sub + created_at (descending)
gcloud firestore indexes composite create \
  --collection-group=personas \
  --query-scope=COLLECTION \
  --field-config field-path=user_sub,order=ascending \
  --field-config field-path=created_at,order=descending

    SQLite (Local Development)

    Used by local Docker Compose stack with Keycloak: - Serverless - In-memory or file-based SQLite database - Schema defined in personadb_sqlite.py - Simple, zero-configuration setup

    Service Architecture

    User-Profile-API

    Purpose: Persona lifecycle management (CRUD operations)

    Endpoints:

    • POST /v1/personas - Create persona
    • GET /v1/personas - List user's personas
    • GET /v1/personas/{persona_id} - Get specific persona
    • PUT /v1/personas/{persona_id} - Update persona
    • DELETE /v1/personas/{persona_id} - Delete persona
    • GET /v1/users/{user_sub}/personas - List personas for any user (service accounts only)

    Authorization:

    • User endpoints: JWT sub must match persona owner
    • Service endpoint: Requires service account token (Keycloak: client_id=flowpilot-agent, GCP: gserviceaccount.com in email)

    Code Location: flowpilot-services/user-profile-api/

    AuthZ-API

    Purpose: Fetches persona data for authorization decisions (acts as Policy Information Point)

    Persona Fetching Logic:

    1. Extract owner.id and owner.persona (title) from AuthZEN request
    2. If owner.persona_id is present, fetch directly: GET /v1/personas/{persona_id}
    3. If only owner.persona (title) is present, fetch by user and title: GET /v1/users/{user_sub}/personas
    4. If fetch fails (404, 403, timeout), use default values (deny by default)
    5. Augment resource.properties.owner with autobook attributes for OPA

    Code Location: flowpilot-services/authz-api/authz_core.py

    Key Functions:

    • fetch_persona_from_api(persona_id) - Fetch by ID
    • fetch_persona_by_user_and_title(user_sub, persona_title) - Fetch by user and title
    • build_opa_input() - Augment OPA input with persona data

    OPA Integration

    Persona data flows into OPA as part of the resource.properties.owner object:

    {
  "subject": {"id": "agent-runner", "persona": "ai-agent"},
  "action": {"name": "execute"},
  "resource": {
    "properties": {
      "owner": {
        "id": "user-uuid",
        "persona": "traveler",
        "persona_id": "b9678f30-f4b0-4033-82db-846357311165",
        "persona_status": "active",
        "persona_valid_from": "2024-01-01T00:00:00Z",
        "persona_valid_till": "2026-12-31T23:59:59Z",
        "autobook_consent": true,
        "autobook_price": 5000,
        "autobook_leadtime": 7,
        "autobook_risklevel": 3
      }
    }
  }
}

    OPA policies use these attributes to evaluate authorization gates:

    # Consent check
has_consent if {
  input.resource.properties.owner.autobook_consent == true
}

# Cost gate
within_cost_limit if {
  planned_price := input.resource.properties.planned_price
  max_cost := input.resource.properties.owner.autobook_price
  planned_price <= max_cost
}

# Persona status check
owner_persona_active if {
  input.resource.properties.owner.persona_status == "active"
}

# Persona temporal validity
owner_persona_valid_time if {
  valid_from_str := input.resource.properties.owner.persona_valid_from
  valid_till_str := input.resource.properties.owner.persona_valid_till
  valid_from := time.parse_rfc3339_ns(valid_from_str)
  valid_till := time.parse_rfc3339_ns(valid_till_str)
  now := time.now_ns()
  now >= valid_from
  now <= valid_till
}

    Persona Validation Rules

    OPA Policy Rules

    The auto_book.rego policy enforces these persona rules:

    Execute Action Rules

    1. Authorized Principal + Persona Validation

      • Owner executing their own workflow: subject.persona must match owner.persona
      • Delegated user: subject.persona must be one of: travel-agent, office-manager, booking-assistant, ai-agent
      • AI agent activated by owner: context.principal.persona must match owner.persona
      • AI agent activated by delegate: context.principal.persona must be an agent persona

    2. Persona Status

      • owner.persona_status must be "active"
      • If missing, defaults to allowing (backward compatibility)

    3. Persona Temporal Validity

      • Current time must be between owner.persona_valid_from and owner.persona_valid_till
      • If missing, defaults to allowing (backward compatibility)

    Read Action Rules

    1. Owner can always read (no persona restrictions)
    2. AI agents can read (needed to list workflow items)
    3. Delegated users with read permission:
      • Must have valid delegation with "read" action
      • subject.persona must be either:
      • An agent persona (travel-agent, office-manager, booking-assistant, ai-agent)
      • OR match owner.persona

    API Validation Rules

    The user-profile-api enforces these constraints:

    • title must be one of the allowed persona titles (configurable via ALLOWED_PERSONA_TITLES env var)
    • scope must be a non-empty array of action strings
    • valid_from & valid_till must be valid ISO 8601 timestamps
    • status must be one of: "active", "inactive", "suspended", "expired"
    • autobook_price must be a non-negative integer
    • autobook_leadtime must be a non-negative integer
    • autobook_risklevel must be an integer between 0 and 100

    Provisioning Personas

    Local Development (Keycloak)

    Use the seed_keycloak_users.py script:

    python3 flowpilot-provisioning/seed_keycloak_users.py \
  --csv flowpilot-provisioning/users_seed.csv

    GCP Production (Firebase)

    Use the seed_firebase_users.py script:

    FIREBASE_API_KEY=<your-api-key> \
python3 flowpilot-provisioning/seed_firebase_users.py \
  --profile-api-url=https://flowpilot-user-profile-api-737191827545.us-central1.run.app \
  --csv=flowpilot-provisioning/users_seed.csv

    CSV Format

    The users_seed.csv file defines users and their personas:

    username;password;email;firstname;lastname;persona;autobook_consent;autobook_price;autobook_leadtime;autobook_risklevel;persona_status;persona_valid_from;persona_valid_till
carlo;password;carlo@me.com;Carlo;Schupp;traveler;Yes;10000;7;5;active;2024-01-01T00:00:00Z;2026-12-31T23:59:59Z
yannick;password;yannick@me.com;Yannick;De Backer;travel-agent,office-manager;Yes;1500;7;2;active;2024-01-01T00:00:00Z;2026-12-31T23:59:59Z

    Note: Users can have multiple personas by comma-separating them in the persona column. Each persona is created independently with the same autobook settings.

    Authorization Scenarios

    Scenario 1: Owner with Traveler Persona

    Context:

    • Carlo (traveler) creates a workflow
    • Carlo executes his own workflow

    Authorization Flow:

    1. Domain-services-api receives request with Carlo's token
    2. Authz-api extracts sub=carlo-uuid from JWT
    3. Authz-api fetches Carlo's "traveler" persona from user-profile-api
    4. OPA evaluates:
      • authorized_principal: ✓ (owner == principal)
      • persona_valid: ✓ (traveler == traveler)
      • owner_persona_active: ✓ (status == "active")
      • owner_persona_valid_time: ✓ (current time within range)
      • has_consent: ✓ (autobook_consent == true)
      • Other gates...
    5. Decision: Allow (if all gates pass)

    Scenario 2: Delegated Travel Agent

    Context:

    • Carlo (traveler) delegates to Yannick (travel-agent)
    • Yannick executes Carlo's workflow

    Authorization Flow:

    1. Domain-services-api receives request with Yannick's token
    2. Authz-api extracts sub=yannick-uuid from JWT
    3. Authz-api queries delegation-api: valid delegation exists
    4. Authz-api fetches Carlo's "traveler" persona (resource owner)
    5. OPA evaluates:
      • authorized_principal: ✓ (valid delegation with "execute" action)
      • persona_valid: ✓ (Yannick's persona "travel-agent" is an agent persona)
      • owner_persona_active: ✓ (Carlo's persona status == "active")
      • owner_persona_valid_time: ✓ (Carlo's persona is valid)
      • has_consent: ✓ (Carlo's autobook_consent == true)
      • Other gates...
    6. Decision: Allow (if all gates pass)

    Scenario 3: Autonomous AI Agent

    Context:

    • AI agent attempts to book autonomously (no delegation)
    • Carlo's workflow with autobook consent

    Authorization Flow:

    1. AI-agent-api calls domain-services-api with service token
    2. AuthZEN request: subject.persona = "ai-agent", context.principal.id = carlo-uuid
    3. Authz-api fetches Carlo's "traveler" persona
    4. OPA evaluates:
      • authorized_principal: ✓ (autobook_consent == true, no delegation required)
      • persona_valid: ✓ (context.principal.persona == owner.persona)
      • owner_persona_active: ✓ (status == "active")
      • owner_persona_valid_time: ✓ (within valid time range)
      • has_consent: ✓ (autobook_consent == true)
      • within_cost_limit: Check Carlo's autobook_price
      • sufficient_advance: Check Carlo's autobook_leadtime
      • acceptable_risk: Check Carlo's autobook_risklevel
    5. Decision: Allow or Deny based on ABAC gates

    Scenario 4: Persona Mismatch

    Context:

    • Martine has two personas: "traveler" and "office-manager"
    • Martine (office-manager) tries to execute her own workflow created with "traveler" persona

    Authorization Flow:

    1. Workflow was created with owner.persona = "traveler"
    2. Martine's request has subject.persona = "office-manager"
    3. OPA evaluates:
      • authorized_principal: ✓ (owner == principal)
      • persona_valid: ✗ (office-manager ≠ traveler, and office-manager is not allowed for owner execution)
    4. Decision: Deny with reason_code "auto_book.persona_mismatch"

    Resolution: Martine must switch to her "traveler" persona when executing her own workflows.

    Troubleshooting

    Persona Not Found

    Symptom: Authorization denied with auto_book.no_consent despite user having consent

    Causes:

    1. Persona not created or provisioned
    2. Persona fetch fails (403, 404, timeout)
    3. Service account cannot access user-profile-api

    Debug Steps:

    1. Verify persona exists: 

      curl -H "Authorization: Bearer <user-token>" \
  https://user-profile-api/v1/personas

    2. Check authz-api logs for persona fetch errors: 

      gcloud logging read "resource.labels.service_name=flowpilot-authz-api \
  AND (textPayload=~'persona' OR jsonPayload.error=~'persona')" \
  --limit=20

    3. Verify service account can access user-profile-api: 

      # Get service token (GCP)
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/identity?audience=flowpilot-services"

# Test access
curl -H "Authorization: Bearer <service-token>" \
  https://user-profile-api/v1/users/<user-sub>/personas

    Persona Status Denied

    Symptom: Authorization denied with auto_book.persona_inactive or auto_book.persona_expired

    Causes:

    1. Persona status is not "active"
    2. Current time is outside valid_from/valid_till range

    Resolution:

    1. Update persona status: 

      curl -X PUT https://user-profile-api/v1/personas/<persona-id> \
  -H "Authorization: Bearer <user-token>" \
  -H "Content-Type: application/json" \
  -d '{"status": "active"}'

    2. Update temporal validity: 

      curl -X PUT https://user-profile-api/v1/personas/<persona-id> \
  -H "Authorization: Bearer <user-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "valid_from": "2024-01-01T00:00:00Z",
    "valid_till": "2027-12-31T23:59:59Z"
  }'

    Firestore Index Missing

    Symptom: User-profile-api returns 500 error: "The query requires an index"

    Resolution: Create the required composite indexes (see "Storage Backends" section above)

    Service Account Authorization Failure

    Symptom: User-profile-api returns 403: "Forbidden: Service account required"

    Causes:

    1. Service token doesn't have client_id=flowpilot-agent (Keycloak)
    2. Service token doesn't have gserviceaccount.com in email (GCP)

    Resolution:

    • Local: Use correct Keycloak service account credentials
    • GCP: Ensure authz-api uses GCP identity tokens from metadata server

    Best Practices

    1. Use descriptive persona titles - Match business roles, not technical identities
    2. Set reasonable autobook limits - Start conservative, adjust based on user comfort
    3. Validate temporal ranges - Ensure valid_till is far enough in the future
    4. Monitor persona status - Implement workflows to expire/suspend personas when needed
    5. Limit personas per user - Most users should have 1-2 personas (configurable via MAX_PERSONAS_PER_USER)
    6. Test delegation with personas - Verify agent personas work correctly with delegations
    7. Use active personas for authorization - Inactive/suspended personas should not be used
    8. Provision test data properly - Use seed scripts to maintain consistency across environments
